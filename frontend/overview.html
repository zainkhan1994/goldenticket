Architecture — 5 Layers

🗄️

1 · Data Layer — Google Cloud Storage

Raw behavioral and health data lives in GCS as Apache Parquet files. Parquet's columnar layout means only the columns you query are read — roughly 50× less I/O than CSV for a 50-column dataset. Per-column compression typically achieves 3–6× size reduction.

Parquet GCS
⚡

2 · Processing Layer — cuDF + UVM on Colab Enterprise

scripts/process_data.py loads the Parquet data using cuDF, NVIDIA's drop-in pandas replacement that runs on the GPU. A single %load_ext cudf.pandas call redirects all pandas operations to the GPU. Unified Virtual Memory (UVM) is enabled via RMM so datasets larger than GPU VRAM spill to CPU RAM instead of crashing.

cuDF NVIDIA RAPIDS UVM
☸️

3 · Orchestration Layer — Google Kubernetes Engine

scripts/setup_gke.sh creates a GKE cluster with two node pools: a default CPU pool for system components and a GPU pool (gpu-pool) that auto-scales from 0 to 1 nodes. The GPU pool only incurs cost (~$0.70/hr for an L4 node, approximate; see GCP pricing) when a GPU pod is scheduled. The cluster control plane (~$0.10/hr, approximate) is always running.

GKE Kubernetes Auto-scaling
🧠

4 · Inference Layer — NVIDIA NIM (Llama 3 8B)

scripts/deploy_nim.sh deploys NIM via Helm. NIM pulls Llama 3 weights from NGC, compiles them with TensorRT for the specific GPU type (layer fusion, FP16 precision, kernel auto-tuning), and caches the engine on a persistent volume. It serves an OpenAI-compatible REST API — no model serving code required.

NVIDIA NIM TensorRT Llama 3 Helm
🌐

5 · Frontend Layer — Browser Chat UI

frontend/index.html is a single HTML file with no build tools or npm dependencies. It maintains conversation history client-side (the API is stateless), pings /v1/models on load to show connection status, and surfaces friendly error messages (e.g., "run kubectl port-forward") when the model is unreachable. Traffic tunnels via kubectl port-forward — no public LoadBalancer needed.

HTML / JS OpenAI API kubectl port-forward
